# Registration Experiment Record

Updated: 2026-05-18

This document records the registration experiments run for MPS-GAF and the
adapted RPM-Net baseline.  The goal is to keep the comparison traceable before
making another model-level change.

## Common Protocol

Unless noted otherwise, the experiments below use the same small training node:

- Dataset: real ModelNet40 HDF5, `modelnet40_ply_hdf5_2048`
- Server dataset path: `datasets/modelnet40_ply_hdf5_2048`
- Noise type: `crop`
- Number of source variants per reference: `2`
- Batch grouping: `groups_per_batch=1`, so each batch contains 2 source-reference pairs sharing one reference
- Training epochs: `50`
- Training limit: `max_train_steps=10`
- Validation limit: `max_eval_batches=5`
- Validation pairs per epoch: `10`
- Learning rate: `1e-4`
- Training iterations: `num_train_iter=2`
- Evaluation iterations: `num_eval_iter=5`
- Seed: `0`
- Device: `cuda:0`

These are controlled comparison runs, not full-scale final test results.

## Data Pipeline Notes

The grouped dataloader is required.  A batch is organized as:

- `points_src`: `groups_per_batch * num_sources_per_ref` rows
- `points_ref`: repeated reference rows aligned with each source row
- `transform_gt`: per-source ground-truth transform
- group metadata that keeps adjacent sources tied to the same reference

For the current protocol, one batch contains 2 rows.  The two source rows are
different transformed/augmented source variants of the same reference object.
Training data is regenerated by epoch seed, while validation/test data remains
deterministic for comparable metrics.

The RPM-Net original implementation was adapted to use the same data loading

## Early MPS-GAF Attempts

These runs were used to locate unstable components before adding explicit
ablation switches.

| Attempt | Setting | Result |
|---|---|---|
| Full fusion from first iteration | Feature normalization + `num_train_iter=2`, fusion active immediately | Unstable and worse than delayed fusion |
| Delayed hand-stat fusion | Full fusion delayed until after iteration 0 | Best Chamfer `0.02413647` at epoch 36; final epoch degraded to `0.03127003` |
| Transform + Chamfer auxiliary loss | `wt_transform=1.0`, `wt_chamfer=1.0`, 20 epochs | Best Chamfer `0.028160`, not helpful |
| Transform + stronger Chamfer loss | `wt_transform=0.5`, `wt_chamfer=2.0`, 20 epochs | Best Chamfer `0.028460`, not helpful |
| Learned target-anchor gate | Learned/content gate without hand statistics, 50 epochs | Best Chamfer `0.025287`, worse than delayed hand-stat fusion |
| Mean-anchor fusion | Mean source-to-reference messages, 20 epochs | Best Chamfer `0.029144`, worse |

Post-hoc gate scans showed that the fusion scale must be very small.  On the
latest delayed-fusion checkpoint, setting the gate logit near `-12` to `-8`
produced lower Chamfer than stronger fusion.  This motivated the later
`fusion_logit_init=-8` experiments.

## Continued Improvement: Self-Correlation and Chamfer-Loss Alignment

After the stabilized run, two low-risk improvement directions were tested.

### Gated Residual Self-Correlation

The self-correlation module had previously improved rotation/translation when
used alone, but direct replacement hurt group-fusion Chamfer.  A residual gate
was added:

- `--self_corr_mode residual`
- `--self_corr_logit_init`
- `--freeze_self_corr_gate`

The residual form injects self-correlation as a small update instead of
replacing the base feature:

```text
feat_corr = normalize(feat + sigmoid(self_corr_logit) * (self_corr(feat) - feat))
```

Results with stabilized training:

| Variant | Epochs run | Best epoch | Best Rot | Best Trans | Best Chamfer | Final Chamfer |
|---|---:|---:|---:|---:|---:|---:|
| baseline stabilized, no self | 92 | 52 | 12.4687 | 0.1657 | **0.02259882** | 0.02554350 |
| self residual gate -8 | 102 | 62 | 14.5074 | 0.1740 | 0.02336361 | 0.02538066 |
| self residual gate -10 | 92 | 52 | 13.2752 | 0.1746 | 0.02296368 | 0.02591480 |
| self residual gate -12 | 92 | 52 | 13.9836 | 0.1587 | 0.02333472 | 0.02555899 |

This did not beat the no-self stabilized baseline.  The self-correlation path is
therefore not the next best direction under this protocol.

## Pose-Oriented Search

A pose-oriented checkpoint metric was added for experiments where rotation and
translation are the priority:

- `--best_metric pose`
- `--pose_trans_weight 50`

The pose selection metric is:

```text
rotation_error_deg_mean + pose_trans_weight * translation_error_mean
```

With `pose_trans_weight=50`, rotation and translation contribute on comparable
scales under the current validation protocol.  LR plateau and early stopping can
therefore follow the pose metric instead of Chamfer.

The best MPS-GAF pose-oriented configuration found so far is:

### SVD Confidence Weighting

The SVD weighting rule was extended beyond RPM-Net's row-sum weighting:

- `row_sum`: original row-sum weights.
- `rowmax`: row-sum multiplied by each source point's maximum correspondence
  probability.
- `entropy`: row-sum multiplied by normalized row-confidence
  `1 - entropy(row)`.
- `mutual`: row-sum multiplied by source-side and matched reference-side
  confidence.

The comparison below uses the same source-2, 200-epoch, 20-eval-batch protocol
and pose selection metric.

| Variant | Best epoch | Rot mean | Trans mean | Chamfer | Pose metric |
|---|---:|---:|---:|---:|---:|
| RPM-Net pose-best | 189 | **10.3604** | **0.0987** | 0.02925128 | **15.2972** |
| `svd_weight_mode=entropy` | 106 | 14.0389 | 0.1682 | 0.02282226 | 22.4476 |
| MPS-GAF current pose baseline | 109 | 14.1658 | 0.1669 | 0.02296519 | 22.5102 |
| `svd_weight_mode=mutual` | 27 | 23.7761 | 0.2641 | 0.02736762 | 36.9793 |
| `mutual`, top-70% | 45 | 23.2270 | 0.2751 | 0.02884703 | 36.9881 |
| `rowmax`, top-70% | 55 | 25.1690 | 0.2536 | 0.02965864 | 37.8447 |
| `rowmax` | 43 | 24.6630 | 0.2730 | 0.02998506 | 38.3144 |

Interpretation:

- Entropy weighting is the only useful SVD-confidence variant so far.  It gives
  a small Rot/pose improvement over the matrix-MSE pose baseline.
- Row-max and mutual-confidence weighting over-prune useful soft
  correspondences under the current partial-overlap protocol and degrade pose.
- SVD weighting alone is not enough to beat RPM-Net's pose-best checkpoint.

### Geometric Refinement

A test-time geometric refinement was added to `mps_gaf_run.py`:

- `--icp_refine_steps`
- `--icp_trim_fraction`
- `--icp_mode point|plane`
- `--icp_damping`
- `--icp_max_angle_deg`
- `--icp_max_translation`

`point` mode is trimmed point-to-point ICP.  `plane` mode is trimmed
point-to-plane ICP using the reference normals from ModelNet40.  The refinement
does not use labels; it only refines the predicted transform against the input
source/reference point sets.

Best MPS-GAF result found so far for Rot/Trans:

```bash
python -u mps_gaf_run.py \
  --mode eval \
  --dataset_path datasets/modelnet40_ply_hdf5_2048 \
  --output_dir runs/mps_geo_w000_p010_plane40_trim0.7_a10 \
  --checkpoint runs/mps_gaf_pose_geo_w000_p010_eval20_src2_200e/mps_gaf_best.pt \
  --noise_type crop \
  --num_sources_per_ref 2 \
  --groups_per_batch 1 \
  --max_eval_batches 20 \
  --num_eval_iter 5 \
  --device cuda:0 \
  --seed 0 \
  --fusion_mode full \
  --fusion_start_iter 1 \
  --no_self_corr \
  --fusion_logit_init -8 \
  --svd_weight_mode row_sum \
  --icp_mode plane \
  --icp_refine_steps 40 \
  --icp_trim_fraction 0.7 \
  --icp_max_angle_deg 10 \
  --icp_max_translation 0.2
```

Results:

Interpretation:

- Against RPM-Net's original output, MPS-GAF with point-to-plane refinement now
  has better rotation and translation.
- If the exact same point-to-plane refinement is also applied to RPM-Net's final
  checkpoint, RPM-Net still has better translation and pose metric.  Therefore
  the honest claim is: MPS-GAF plus geometric refinement beats the original
  RPM-Net output, but not RPM-Net with the same extra geometric refinement.
- The refinement improves pose at the cost of Chamfer.  This is expected under
  partial overlap because point-to-plane ICP optimizes local surface alignment
  rather than symmetric full Chamfer.

### Train-Time Geometry and Correspondence Losses

Two internal losses were added to test whether the model can learn the geometric
correction that point-to-plane ICP provides at evaluation time:

- `--wt_plane`: trimmed point-to-plane residual between transformed source
  points and nearest reference surface normals.
- `--wt_corr`: overlap-aware correspondence supervision.  The GT-transformed
  source is matched to the nearest reference point; points within
  `--corr_overlap_radius` are supervised toward that reference index, and
  non-overlap points are encouraged to have low row mass.

These losses do not introduce any external model or pretrained component.  They
only use the existing point sets, normals, predicted permutation matrix, and
available training transforms.

Scout protocol:

- ModelNet40 HDF5, `crop`
- `num_sources_per_ref=2`
- `groups_per_batch=1`
- `max_train_steps=10`
- `max_eval_batches=20`
- 120 epochs, `best_metric=pose`, `pose_trans_weight=50`

Raw-output validation results:

|---|---:|---:|---:|---:|
| Previous entropy-SVD pose checkpoint | **14.0389** | **0.1682** | 0.02282226 | **22.4476** |
| `wt_pose=0.10`, `wt_plane=50`, `wt_corr=0.01` | 14.1406 | 0.1969 | 0.02298872 | 23.9878 |
| `wt_chamfer=0.10`, `wt_transform=0.10`, entropy SVD, `wt_plane=50`, `wt_corr=0.01` | 15.3581 | 0.2182 | **0.02244407** | 26.2703 |

### Learned SVD Inlier Weighting

To make the improvement more learning-based, a learnable point-wise inlier head
was added before the weighted SVD step.  The head predicts each source point's
SVD reliability from:

- source feature;
- soft-matched reference feature;
- absolute feature residual;
- Sinkhorn row mass;
- row maximum probability;
- row entropy confidence;
- current geometric residual to the soft-matched reference point.

The learned head is initialized to approximate the old SVD weighting, so it does
not start by randomly changing the rigid solver.  It can be trained with the
existing pose/Chamfer losses and optional overlap supervision:

- `--svd_weight_mode learned`
- `--svd_weight_mode learned_entropy`
- `--wt_svd_inlier`
- `--svd_inlier_radius`
- `--svd_inlier_pos_weight`

Best learned configuration so far:

  --dataset_path datasets/modelnet40_ply_hdf5_2048 \

  --checkpoint runs/mps_gaf_pose_svd_entropy_eval20_src2_200e/mps_gaf_best.pt \
  --noise_type crop \
  --num_sources_per_ref 2 \
  --groups_per_batch 1 \
  --epochs 180 \
  --max_train_steps 10 \
  --max_eval_batches 20 \
  --lr 5e-5 \
  --num_train_iter 2 \

  --device cuda:0 \
